- Gestion d'erreurs : Now let me check the relevant source files for context on error handling patterns.# Review — Gestion d'erreurs

Le code nouveau est essentiellement de la présentation SwiftUI (views pures) + une propriété exposée dans le store. La surface d'erreur est réduite, mais voici les points notables :

1. **`TimeWindowMetricRow.swift:47` — `DateFormatter` alloué à chaque calcul de `body`**
   `resetDateText` (et `theoreticalFraction`, `remainingText`) instancient un `DateFormatter` / calculent des dates à chaque rendu. Ce n'est pas un bug d'erreur mais c'est une fragilité de thread-safety documentée dans `docs/SWIFT-CONCURRENCY.md` : `DateFormatter` n'est pas thread-safe. Ici c'est un `struct View` recalculé sur le main thread, donc acceptable — mais si ce code est extrait un jour dans un contexte concurrent, il cassera. Considérer un `static let` formatter par locale ou utiliser `ISO8601DateFormatter` déjà en place dans `ISODate`.

2. **`TimeWindowMetricRow.swift:21` — `Date()` direct (non injectable)**
   `theoreticalFraction` et `remainingText` appellent `Date()` directement. Cela rend ces propriétés calculées impossibles à tester de manière déterministe (le résultat dépend de l'heure d'exécution). Selon `docs/SWIFT-TESTABILITY.md`, le clock devrait être injectable. **Recommandation** : accepter un `now: Date` en paramètre ou un `Clock` protocol pour permettre des tests unitaires fiables sur les calculs de temps restant / fraction théorique.

3. **`PayAsYouGoMetricRow.swift:10-17` — `NumberFormatter` fallback silencieux**
   Le fallback `"\(currency) \(String(format: "%.2f", currentAmount))"` est bien présent si `formatter.string(from:)` retourne `nil`. C'est correct — le chemin d'erreur est couvert. Aucun problème ici.

4. **`TimeWindowMetricRow.swift:17,26,46` — `ISODate.date` retourne `nil` sur parsing invalide**
   Trois propriétés calculées gèrent le cas `resetAt.date == nil` avec des valeurs de fallback (`return 0`, `return "0m"`, `return "--"`). Les chemins sont couverts. Cependant, l'échec de parsing est **silencieux** : l'utilisateur verra "0m" et "--" sans savoir que la date est invalide. **Recommandation mineure** : pour `remainingText`, afficher `"--"` plutôt que `"0m"` quand le parsing échoue — `"0m"` est trompeur (il suggère que le reset est imminent alors que la date est simplement non parsable).

5. **`UsageDetailsView.swift:20` — `ForEach` avec `id: \.offset` au lieu d'un identifiant stable**
   Utiliser l'index comme identité SwiftUI peut provoquer des animations incohérentes et des bugs de recycling si la liste change pendant un rendu. Si `VendorUsageEntry` n'est pas `Identifiable`, il devrait le devenir (avec un id composite `vendor + account`). Pas un bug d'erreur à proprement parler, mais un cas limite de UI qui peut causer des comportements inattendus lors de mises à jour rapides.

6. **`UsageStore.swift:115` — `refreshMenuBarText()` ré-assigne `entries` depuis `lastFile.usages`**
   Ce n'est pas un bug, mais chaque tick du countdown timer reconstruit le tableau `entries` même si les données n'ont pas changé. Avec beaucoup d'entries, cela déclenchera des re-rendus SwiftUI inutiles. **Recommandation** : comparer avant de ré-assigner, ou ne mettre à jour `entries` que dans `handleFileData`.

**Résumé** : Pas de silent swallowing critique ni de chemin d'erreur non couvert. Le point le plus actionnable est le **#4** (`"0m"` trompeur quand la date est invalide) et le **#2** (testabilité du clock).

## Tests existants — faiblesses

8. **`UsageStoreTests.swift:495-496`** — `multipleEntries` asserte l'ordre des entries par index (`entries[0].isActive == true`, `entries[1].isActive == false`) mais l'ordre dépend de l'ordre du JSON, pas d'un tri garanti. Si l'implémentation change l'ordre, le test casse pour une mauvaise raison. Mieux vaut tester avec `contains` ou trier explicitement.

C'est le manque le plus critique car c'est la logique métier la plus complexe de cette PR et elle est entièrement non couverte.

## Principe non-négociable — discipline des commits

**Tu commites AVANT de présenter le résultat à l'utilisateur, JAMAIS
après son approbation finale.** Dans cette phase : l'utilisateur a
déjà approuvé une *sélection* de recommandations → tu les appliques
→ tu commites → tu présentes le résultat → l'utilisateur choisit
ensuite entre "apply more", "re-run reviews" ou "ship it". Le
commit correspond à un état produit, pas à une validation finale.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception.**

**FIXUP — règle absolue pour cette phase :** chaque application de
recommandation doit produire des commits `--fixup` rattachés au commit
source qui a introduit le code modifié, PAS des commits indépendants
`review(axe):`. Procédure pour chaque bloc de fichiers modifiés :
```bash
# Trouver le commit source pour les fichiers concernés
git log --oneline origin/main..HEAD -- <fichier1> <fichier2>
# Le commit le plus récent qui a touché ces fichiers est le commit source
SOURCE_HASH=<hash du commit source>
git add <fichiers du bloc>
git commit --fixup=$SOURCE_HASH
```
Si un bloc touche des fichiers issus de **plusieurs** commits sources
distincts, crée **un fixup par commit source** (staging séparé par commit).
L'historique final doit être squashable proprement avec
`git rebase --autosquash`.

## Principe non-négociable — discipline des commits

**Tu commites AVANT de présenter le résultat à l'utilisateur, JAMAIS
après son approbation finale.** Dans cette phase : l'utilisateur a
déjà approuvé une *sélection* de recommandations → tu les appliques
→ tu commites → tu présentes le résultat → l'utilisateur choisit
ensuite entre "apply more", "re-run reviews" ou "ship it". Le
commit correspond à un état produit, pas à une validation finale.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception.**

**FIXUP — règle absolue pour cette phase :** chaque application de
recommandation doit produire des commits `--fixup` rattachés au commit
source qui a introduit le code modifié, PAS des commits indépendants
`review(axe):`. Procédure pour chaque bloc de fichiers modifiés :
```bash
# Trouver le commit source pour les fichiers concernés
git log --oneline origin/main..HEAD -- <fichier1> <fichier2>
# Le commit le plus récent qui a touché ces fichiers est le commit source
SOURCE_HASH=<hash du commit source>
git add <fichiers du bloc>
git commit --fixup=$SOURCE_HASH
```
Si un bloc touche des fichiers issus de **plusieurs** commits sources
distincts, crée **un fixup par commit source** (staging séparé par commit).
L'historique final doit être squashable proprement avec
`git rebase --autosquash`.
