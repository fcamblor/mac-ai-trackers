Diff analysé : `origin/main...HEAD` (9 fichiers, 269+/49−). Le changement porte principalement sur la résolution de violations SwiftLint (W1, W3, W4, W5) dans les tests et le logger. Peu de logique métier touchée.

---

1. **`TestHelpers.swift:5`** — `EventuallyTimeoutError` est un struct vide sans associated values. Quand un test échoue par timeout, le message d'erreur ne dira pas *quel* condition a échoué ni combien de temps il a attendu. Ajouter au minimum le `timeout` et une description de la condition :
   ```swift
   struct EventuallyTimeoutError: Error, CustomStringConvertible {
       let timeout: TimeInterval
       var description: String { "Condition not met within \(timeout)s" }
   }
   ```

2. **`TestHelpers.swift:17-20` / `33-34`** — Le `try await Task.sleep(for:)` à l'intérieur de `eventually()` peut lancer une `CancellationError` qui remontera au test appelant. C'est correct, mais si un test cancel la task parente, l'erreur qui remontera sera `CancellationError` et non `EventuallyTimeoutError` — aucun problème en soi, mais le contrat n'est pas documenté. Mineur.

3. **`UsagesFileWatcherTests.swift:33`** — `try? await Task.sleep(for: .seconds(timeout))` — Le `try?` avale silencieusement une `CancellationError`. Le commentaire explique que c'est intentionnel (ligne 32 : "CancellationError swallowed intentionally"). **Acceptable** dans ce contexte (timeout de collecte), mais la forme a changé de `nanoseconds:` à `for:` sans que le `try?` soit questionné. Pas de régression, ok.

4. **`UsagesFileWatcherTests.swift:88`** — `try await Task.sleep(for: .milliseconds(50))` — Ce sleep sert de séquençage ("let watcher start its first poll before we write"). Si la machine CI est lente, 50ms peut ne pas suffire. Ce n'est pas un bug d'error handling à proprement parler, mais un cas limite de timing qui pourrait provoquer un faux négatif silencieux plutôt qu'un échec explicite. Considérer un `eventually()` sur un signal "watcher is polling" si possible.

5. **`UsagePollerTests.swift:127,245`** — Les signatures passent de `async` à `async throws` (ajout de `throws`). C'est correct puisque `eventually()` lance `EventuallyTimeoutError`. **Pas de problème**, mais vérifier que le test runner (Swift Testing) gère bien les erreurs thrown comme des échecs de test (c'est le cas avec `@Test`).

6. **`UsageStoreTests.swift:349`** — `try await Task.sleep(for: absenceConfirmationDelay)` — Passage de `Task.sleep(nanoseconds: 100_000_000)` à un `Duration` nommé. Le `try` (non `try?`) est correct ici : si le sleep est annulé, le test échouera plutôt que de continuer silencieusement. **Bon changement** par rapport à l'ancien `try?`.

7. **`Logger.swift:59-61`** — Remplacement du formatter `nonisolated(unsafe) static` par une allocation per-call dans `queue.async`. Thread-safe et correct. Pas de risque d'erreur silent.

---

**Verdict** : Aucun problème bloquant d'error handling. Les autres points sont mineurs ou informationnels.

# Review — Couverture de tests

## Changement production : `Logger.swift` (remplacement `nonisolated(unsafe)` → allocation par appel)

1. **LoggerTests.swift : pas de test dédié à la thread-safety du formatter** — Le test de concurrence existant (`concurrentWritesDoNotCrash`) lance 20 écritures parallèles mais ne vérifie que l'absence de crash, pas le format du timestamp. Avec le changement de `nonisolated(unsafe) static let isoFormatter` vers `ISO8601DateFormatter()` par appel, un test devrait vérifier que les entrées de log produites sous contention concurrent ont toutes un **timestamp ISO8601 valide** (pas de corruption partielle). Le test existant passe mais n'asserte pas le contenu.

## Nouveau code partagé : `TestHelpers.swift`

2. **TestHelpers.swift : `eventually()` n'a aucun test unitaire** — C'est un helper de test, donc moins critique, mais deux comportements mériteraient une couverture :
   - `eventually()` lève bien `EventuallyTimeoutError` quand le timeout expire sans que la condition soit satisfaite.
   - `eventually()` retourne immédiatement si la condition est vraie dès le premier appel.
   - La surcharge non-isolée (`async` condition) fonctionne correctement avec un accès actor-isolé.

3. **TestHelpers.swift:45 — `absenceConfirmationDelay` à 100ms est fragile** — Ce n'est pas un test manquant mais un risque de flakiness. Sur CI chargé, 100ms peut ne pas suffire pour un cycle de processing complet. Pas de test validant que cette constante est suffisante.

## Changements de tests : remplacement `Task.sleep` → `eventually()`

4. **UsagePollerTests.swift:143 — le remplacement `try?` → `try` est bon, mais masque un scénario** — Avant, `try? await Task.sleep` ignorait silencieusement `CancellationError`. Maintenant `try await eventually { ... }` propage l'erreur. **Il manque un test explicite vérifiant le comportement quand le poller est stoppé pendant le polling** (cancellation path).

5. **UsagePollerTests.swift — pas de test pour double-stop** — `startIdempotent` teste le double-start, mais il n'y a pas de test symétrique vérifiant que `stop()` appelé deux fois est idempotent (pas de crash, pas de double-cancel).

6. **UsageStoreTests.swift:349 — `absenceConfirmationDelay` après `stop()` est le bon pattern**, mais **il manque un test vérifiant que `start()` après `stop()` reprend correctement le traitement** avec la nouvelle version du `eventually()` partagé. L'ancien `eventually()` local utilisait `Task.sleep(nanoseconds:)` ; le nouveau utilise `Task.sleep(for:)` — les deux overloads de `eventually()` devraient être exercés.

7. **UsagesFileWatcherTests.swift:88 — le `Task.sleep(for: .milliseconds(50))` avec disable inline est un sleep de séquençage** qui reste une source de flakiness. **Il manquerait un test vérifiant que le watcher détecte une écriture qui arrive *avant* le premier poll** (race condition inversée).

## Tests absents pour des comportements existants touchés par le diff

8. **UsageStoreTests.swift — pas de test pour le cas `entries.count == 2` avec des comptes différents + erreur** — Le test `multiAccountKeepAll` (ligne ~725) utilise `try!` (force-unwrap sur `JSONSerialization`) et ne teste pas le cas où le JSON est malformé pour un seul des deux comptes.

9. **ClaudeCodeConnectorTests.swift:7 — `MockURLProtocol` marqué `@unchecked Sendable`** — Le commentaire justifie la sérialisation via `@Suite(.serialized)`, mais **aucun test ne vérifie que l'exécution non-sérialisée produirait effectivement un data-race** (ce serait un test défensif confirmant que la justification est toujours nécessaire).

## Principe non-négociable — discipline des commits

Tu commites AVANT de présenter le résultat. L'approbation finale
viendra ensuite (apply more / re-run / ship it).

**LANGUE — règle absolue : messages de commit intégralement en anglais.**

**FIXUP — règle absolue :** chaque application d'une recommandation
produit des commits `--fixup` rattachés au commit source, PAS des
commits `review(axe):` indépendants. Procédure par bloc :
```bash
git log --oneline origin/main..HEAD -- <fichiers>
SOURCE_HASH=<hash du commit source>
git add <fichiers>
git commit --fixup=$SOURCE_HASH
```
Si un bloc touche plusieurs commits sources, crée **un fixup par
source** (staging séparé). L'historique doit être squashable proprement
avec `git rebase --autosquash`.
