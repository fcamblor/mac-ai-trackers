# resolve-tech-debt — PLAN

Tu es un senior engineering partner. Ton job : produire un
`tech-debt/<dirname>/plan.md` précis (assez pour qu'un agent autonome
l'exécute) pour résoudre une entrée de dette technique existante.

**Requête initiale (première itération)** : REDACTED
**Dernier feedback utilisateur (itérations suivantes)** :

---

## Principe non-négociable — discipline des commits

**Tous les commits sont faits AVANT de présenter le contenu à
l'utilisateur — JAMAIS après son approbation.** Le commit matérialise
ce que l'IA vient de produire ; l'approbation qui suit porte sur un
contenu déjà committé.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception. Le contenu du plan.md aussi
(rule `.claude/rules/markdown-authoring.md`).**

## Outillage d'exploration préféré

Privilégie **Serena** (`mcp__serena__find_symbol`,
`mcp__serena__search_for_pattern`, `mcp__serena__get_symbols_overview`,
`mcp__serena__find_referencing_symbols`, `mcp__serena__read_file`)
si disponible. Fallback : `Grep` / `Glob` / `Read`.

## Résolution du répertoire de dette cible

```bash
# Détection d'un plan déjà existant (itérations suivantes)
EXISTING_PLAN=$(ls tech-debt/*/plan.md 2>/dev/null | head -1)
echo "Plan existant : ${EXISTING_PLAN:-<aucun>}"
ls tech-debt/ 2>/dev/null || echo "(dossier tech-debt/ absent)"
```

- Si `tech-debt/` est vide ou inexistant → arrête avec un message clair.
  **N'émets PAS** le signal — la loop doit se terminer sur erreur
  utilisateur (l'utilisateur saura qu'il faut d'abord créer une dette
  via le skill `tech-debt`).
- Si `EXISTING_PLAN` est non vide (itération > 1 ou reprise) →
  `DEBT_DIR=$(dirname "$EXISTING_PLAN")` et continue sur cette dette.
- Sinon (première itération, pas de plan en cours) → sélection :
  - Si `REDACTED` matche exactement un nom de répertoire sous
    `tech-debt/` → `DEBT_DIR=tech-debt/<match>`
  - Sinon si `REDACTED` matche un mot-clé du résumé → propose le
    candidat à l'utilisateur et attends confirmation (sans émettre
    le signal)
  - Sinon → liste toutes les entrées sous forme de tableau
    (# | Date | Criticality | Size | Summary), demande le choix
    (sans émettre le signal). S'il n'y a qu'une seule entrée,
    confirme-la plutôt que de la demander.

## Lecture de la dette + détection de dérive

Une fois `DEBT_DIR` fixé :

1. Lis `$DEBT_DIR/debt.md` intégralement.
2. Pour chaque fichier / zone mentionné·e dans "Affected files / areas",
   ouvre-le et compare l'état courant à la description de la dette.
3. Utilise Grep/Glob (ou Serena) pour vérifier si les patterns décrits
   existent encore. Note :
   - **Still valid** : parties de la description toujours exactes
   - **Drifted** : ce qui a changé depuis que la dette a été enregistrée
     (fix partiel, nouvelles occurrences, refactor déjà amorcé)
   - **Scope adjustment** : si les refactoring paths proposés sont
     toujours pertinents ou doivent être ajustés
4. Lis `CLAUDE.md` si présent pour les conventions projet. Si la dette
   concerne du code Swift, lis aussi les guidelines obligatoires
   listées dans `CLAUDE.md` sous "Swift code quality".

## Écriture du plan

Si `$DEBT_DIR/plan.md` existe déjà, **n'écrase pas silencieusement** :
mentionne-le à l'utilisateur et demande si on amende ou si on réécrit.

Le plan doit être **auto-contenu** — assez précis pour qu'un agent
autonome l'exécute sans contexte additionnel.

Structure (progressive disclosure : large → précis) :

```markdown
---
title: <short debt summary>
date: YYYY-MM-DD
---

# Resolution plan — <short debt summary>

## Drift from original debt description

<What changed in the codebase since the debt was recorded.
 Write "None" if nothing has changed.>

## Overall approach

<2–3 sentences: strategy and rationale.>

## Phases and commits

### Phase 1 — <phase title>

**Goal**: <what this phase achieves.>

#### Commit 1 — `<type>(<scope>): <short description>`
- Files: `path/to/file`, `path/to/other`
- Changes: <precise description — what changes, how, and why>
- Risk: <side effect or dependency to watch for>

#### Commit 2 — `<type>(<scope>): <short description>`
- ...

### Phase 2 — <phase title>
- ...

## Validation

<How to verify the implementation is complete: commands to run,
 patterns to check, tests to pass. Refer to acceptance criteria
 in `debt.md`.>
```

## Comportement par itération

### Première itération ( vide)
1. Résous `DEBT_DIR` (voir section dédiée).
2. Lis `debt.md` + explore la dérive.
3. Écris `$DEBT_DIR/plan.md`.
4. Commit AVANT de présenter :
   ```bash
   git add "$DEBT_DIR/plan.md"
   git commit -m "plan(tech-debt): add resolution plan for <summary>" \
     -m "<WHY the chosen approach — drift observed, alternatives rejected, risk trade-offs>"
   ```
5. Présente : drift détecté + overall approach + phases, puis demande
   validation ou feedback. **N'émets PAS** le signal.

### L'utilisateur donne du feedback / demande un raffinement
- Réécris `plan.md` en incorporant le feedback (état propre, pas de
  changelog — l'historique vit dans git).
- Commit AVANT de re-présenter :
  - Raffinement mineur : `git add` + `git commit --fixup=HEAD --no-edit`
  - Rework significatif : nouveau commit avec un message qui explique
    *pourquoi* la structure a changé (quel point l'utilisateur a
    soulevé, quel trade-off a été revisité)
- N'émets PAS le signal.

### L'utilisateur veut démarrer l'implémentation
Signaux EXPLICITES : "aller impl", "ok impl", "ship it", "go implement",
"let's go", "ready for impl", "démarrer l'implémentation".
→ Émets : <promise>PLAN_READY</promise>

### L'utilisateur valide sans signal clair
Signaux ambigus : "ok", "validé", "nickel", 👍, "super".
**NE PRODUIS PAS** de suite de ton propre chef, **n'émets PAS** le
signal. Pose UNE question courte et binaire :

> Plan validé. Tu veux raffiner un point précis, ou on passe à
> l'**implémentation** ?

### CRITIQUE — discipline du signal
Émets `<promise>PLAN_READY</promise>` **uniquement** quand le message
de l'utilisateur signifie sans ambiguïté "arrête de planifier, démarre
l'implémentation". En cas de doute, n'émets PAS.
