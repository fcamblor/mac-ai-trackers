# Review — Gestion d'erreurs

1. **`UsagesFileWatcher.swift:53`** — `try?` silencieux sur `attributesOfItem(atPath:)`. Si le fichier existe mais que la lecture des attributs échoue (permissions, volume démonté), l'erreur est avalée et aucun log n'est produit. Le consommateur ne sait jamais pourquoi il ne reçoit plus de mises à jour. Remplacer par un `do/catch` avec un log d'avertissement (au moins la première fois ou avec throttling).

2. **`UsagesFileWatcher.swift:59`** — `FileManager.default.contents(atPath:)` retourne `nil` silencieusement si la lecture échoue. Même situation : la modDate est mise à jour (`lastModDate = modDate`) mais aucune donnée n'est émise et aucun log n'est produit. Pire : puisque `lastModDate` est déjà avancé, la prochaine tentative de poll ignorera ce fichier jusqu'à la prochaine écriture. Il faudrait soit ne pas avancer `lastModDate` quand `contents` retourne `nil`, soit logger l'échec.

3. **`UsagesFileWatcher.swift:90`** — `try? await Task.sleep(...)`. Le `CancellationError` est avalé. C'est intentionnel ici (le `guard !Task.isCancelled` suit), mais c'est fragile : si `Task.sleep` lance une erreur autre que `CancellationError` dans une future version du runtime, elle sera ignorée. Mineur, acceptable.

4. **`UsagesFileWatcher.swift:52-63`** — `emitIfChanged()` est appelé depuis le `setEventHandler` d'un `DispatchSource` (thread GCD utility) **et** depuis le `Task` (cooperative pool). Aucune synchronisation ne protège `lastModDate`. C'est une **race condition** : deux lectures concurrentes peuvent toutes deux voir `modDate != lastModDate`, produisant un doublon d'émission, ou un lecteur peut écraser la date de l'autre. Protéger avec un verrou (`NSLock`), un actor dédié, ou déplacer `emitIfChanged` dans un contexte sérialisé.

5. **`UsageStore.swift:83`** — `try? await Task.sleep(nanoseconds: refreshNanos)`. Même pattern que le point 3 — `CancellationError` avalé. Acceptable mais le commentaire « silent swallow is intentional for cancellation » serait bienvenu.

6. **`UsageStore.swift:143-146`** — `ISO8601DateFormatter` instancié à chaque appel de `formatRemainingTime`. Ce n'est pas un problème d'erreur mais de thread-safety : `ISO8601DateFormatter` (hérite de `Formatter`/`NSFormatter`) n'est pas thread-safe. Ici le `@MainActor` protège, donc OK dans le contexte actuel, mais si `format` est un jour appelé hors main actor, c'est un crash potentiel. Mentionner la contrainte dans un commentaire ou stocker le formatter comme propriété `@MainActor`.

7. **`UsageStore.swift:144`** — Le parsing d'une date ISO 8601 invalide retourne silencieusement `"0m"` sans aucun log. L'utilisateur voit un countdown à zéro sans comprendre que la source de données est corrompue. Logger au moins un warning.

8. **`UsagesFileWatcher.swift:66-77`** — Si `open()` échoue (ligne 67, `fd < 0`), l'erreur POSIX (`errno`) n'est ni loggée ni rapportée. Le watcher tombe silencieusement en mode polling-only sans diagnostic. Logger `errno`/`strerror` quand `open` échoue.

9. **`UsagesFileWatcher.swift:82-86`** — Si `openSource()` retourne `nil` au démarrage (fichier absent, permissions…), aucun log n'est émis. On sait seulement qu'on est en polling via l'absence du log debug. Ajouter un log expliquant pourquoi le FS watcher n'a pas pu démarrer.

10. **`AIUsagesTrackersApp.swift:48`** — `store.start()` est appelé dans un `Task { @MainActor in ... }` séparé du `init()`. Si l'app affiche la menubar avant que ce Task ne s'exécute, `menuBarText` sera `"--"` brièvement — c'est le fallback voulu, donc OK. Mais si le `Task` est retardé (thread contention), le poller et le file watcher pourraient écrire des données avant que le store ne soit en écoute, perdant le premier événement. Le poll timer du `UsagesFileWatcher` finira par rattraper, mais il y a un fenêtre de `pollInterval` (30s) où la menubar montre `"--"` alors que des données existent.

## Display pipeline

A file watcher observes `usages.json` using a hybrid strategy: a kernel
`DispatchSource` fires on `.write`/`.delete`/`.rename` events, backed by a
polling timer that re-checks every 30 seconds to handle atomic replaces and

## Display pipeline

A file watcher observes `usages.json` using a hybrid strategy: a kernel
`DispatchSource` fires on `.write`/`.delete`/`.rename` events, backed by a
polling timer that re-checks every 30 seconds to handle atomic replaces and
network-mount edge cases. Events are debounced to coalesce rapid successive
writes into a single read.

An `@Observable` store (running on `@MainActor`) receives each new file
snapshot, decodes it, locates the active Claude account, and formats the
session and weekly time-window metrics into the menubar label
(`S 48% 2h13m | W 7% 6d 6h 13m`). A secondary timer refreshes the countdown
display every 60 seconds so remaining-time values stay current between file
changes. When data is unavailable or malformed the label falls back to `"--"`.

## Principe non-négociable — discipline des commits

**Tu commites AVANT de présenter le résultat à l'utilisateur, JAMAIS
après son approbation finale.** Dans cette phase : l'utilisateur a
déjà approuvé une *sélection* de recommandations → tu les appliques
→ tu commites → tu présentes le résultat → l'utilisateur choisit
ensuite entre "apply more", "re-run reviews" ou "ship it". Le
commit correspond à un état produit, pas à une validation finale.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception.**

**FIXUP — règle absolue pour cette phase :** chaque application de
recommandation doit produire des commits `--fixup` rattachés au commit
source qui a introduit le code modifié, PAS des commits indépendants
`review(axe):`. Procédure pour chaque bloc de fichiers modifiés :
```bash
# Trouver le commit source pour les fichiers concernés
git log --oneline origin/main..HEAD -- <fichier1> <fichier2>
# Le commit le plus récent qui a touché ces fichiers est le commit source
SOURCE_HASH=<hash du commit source>
git add <fichiers du bloc>
git commit --fixup=$SOURCE_HASH
```
Si un bloc touche des fichiers issus de **plusieurs** commits sources
distincts, crée **un fixup par commit source** (staging séparé par commit).
L'historique final doit être squashable proprement avec
`git rebase --autosquash`.
