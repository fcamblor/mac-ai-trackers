# FCR-Dev — Agrégation des reviews & application de la sélection

## Reviews sources
- Qualité de code : # Review — Qualité de code

1. **[ClaudeCodeConnector.swift:158-160]** `Int(sessionUtil)` tronque le `Double` — si l'API renvoie une utilisation en fraction (ex. `0.75` pour 75 %), le résultat sera `0`. Vérifier si la valeur doit être multipliée par 100 avant la conversion, ou utiliser `Int((sessionUtil * 100).rounded())`.

2. **[UsagePoller.swift:29-34]** `[weak self]` suivi de `guard let self else { return }` **avant** le `while` : une fois le guard passé, `self` est fortement retenu pour toute la durée de la boucle. Le `weak` ne sert à rien. Déplacer le `guard let self` à l'intérieur du `while` pour que la référence forte soit relâchée entre chaque itération et que l'acteur puisse être libéré.

3. **[Logger.swift:39]** Les paramètres `file` et `line` sont capturés (`#file`, `#line`) mais jamais utilisés dans le format du log (l.42). Soit les intégrer dans `entry`, soit les retirer de la signature pour ne pas induire en erreur l'appelant.

4. **[Logger.swift:41]** `ISO8601DateFormatter()` est instancié à chaque appel de `log()`. C'est un objet lourd. Pour un logger potentiellement appelé très fréquemment, en faire une propriété `static let` (ou `nonisolated(unsafe) static let` pour la Sendability).

5. **[UsagesFileManager.swift:74-85]** L'écriture « atomique » est fragile : `moveItem` échoue si la destination existe, puis on `removeItem` + retry. Entre le remove et le move, un autre process peut écrire. Utiliser `Data.write(to:options: .atomic)` directement, ou `FileManager.replaceItemAt(_:withItemAt:)` qui gère le remplacement de façon atomique et sûre.

6. **[UsagesFileManager.swift:39]** `merge(existing:incoming:)` est `public` alors qu'il s'agit d'un détail d'implémentation interne (seul `update(with:)` l'appelle). Il n'est public que pour les tests — préférer un accès `internal` (ou `package`) et `@testable import` suffit déjà dans les tests.
- Gestion d'erreurs : # Review — Gestion d'erreurs

1. **ClaudeCodeConnector.swift:108-116** — `fetchOAuthToken()` appelle `process.waitUntilExit()` sans timeout. Si le processus `security` se bloque (prompt Keychain interactif, deadlock), le thread est bloqué indéfiniment. Ajouter un `DispatchQueue.global().asyncAfter` ou utiliser `terminationHandler` avec un timeout pour tuer le processus après N secondes.

2. **ClaudeCodeConnector.swift:120-121** — `readDataToEndOfFile()` est appelé **après** `waitUntilExit()`. Si le pipe buffer est plein (> 64 Ko), `waitUntilExit()` deadlock car le processus enfant est bloqué en écriture tandis que personne ne lit le pipe. Lire le pipe **avant** d'attendre la fin du processus, ou lire de manière asynchrone.

3. **ClaudeCodeConnector.swift:158** — `Int(sessionUtil)` tronque un `Double` (ex. `0.75` → `0`). Si l'API renvoie une utilisation entre 0.0 et 1.0 (un ratio), le pourcentage affiché sera toujours 0 ou 1. Vérifier le format de l'API ; probablement il faut `Int(sessionUtil * 100)` ou `Int(sessionUtil.rounded())`.

4. **UsagesFileManager.swift:18** — `try? FileManager.default.createDirectory(...)` avale silencieusement l'échec de création du répertoire. Si le dossier n'est pas créé, tous les writes échoueront ensuite sans diagnostic clair à la racine. Logger l'erreur.

5. **UsagesFileManager.swift:76** — `moveItem(atPath:toPath:)` échoue si la destination existe déjà (comportement documenté de `FileManager`). Le fallback (L79-81) supprime le fichier existant puis re-tente le move, mais entre le `removeItem` et le `moveItem`, un crash ou un autre processus concurrent crée une fenêtre de perte de données. Utiliser `replaceItemAt(_:withItemAt:)` qui est atomique sur macOS.

6. **UsagesFileManager.swift:30-34** — `withFlockExclusive` dans `update(with:)` ne retourne rien, mais si le flock échoue (retourne `nil` dans `withFlock`), l'update est silencieusement ignorée : les données reçues sont perdues sans aucun log au niveau appelant. Propager l'échec ou au minimum logger au niveau `update()`.

7. **UsagesFileManager.swift:100-102** — TOCTOU race sur le fichier lock : `fileExists` (L101) suivi de `createFile` (L102) n'est pas atomique. Un autre processus peut supprimer le fichier entre les deux. Utiliser directement `open(lockPath, O_RDWR | O_CREAT, 0o644)` qui crée atomiquement.

8. **UsagePoller.swift:29-35** — Le `Task` capture `[weak self]` mais l'actor `UsagePoller` n'est pas une classe — c'est un actor, qui ne supporte pas `weak`. Ce code compile mais `weak self` n'a aucun effet : l'actor ne sera jamais déalloué tant que la task tourne. `stop()` annule la task, mais si `stop()` n'est jamais appelé (ex. crash avant), la task et l'actor fuient.

9. **UsagePoller.swift:33** — `try? await Task.sleep(for: self.interval)` avale le `CancellationError`. C'est intentionnel (boucle while), mais après le sleep le `Task.isCancelled` est vérifié uniquement au prochain tour de boucle, ce qui signifie qu'un `pollOnce()` complet sera exécuté après l'annulation si le timing est défavorable. Vérifier `Task.isCancelled` immédiatement après le sleep.

10. **UsagePoller.swift:53** — `Loggers.app.log(...)` est utilisé au lieu de `self.logger`. Cela contourne l'injection de dépendance du logger, rendant cette erreur invisible dans les tests qui injectent un logger custom.

11. **Logger.swift:48-61** — `append()` n'a aucune synchronisation. `FileLogger` est marqué `Sendable` et peut être utilisé depuis plusieurs threads/actors concurrents. Deux appels concurrents à `append()` peuvent interleaver les écritures (seek + write non-atomique), corrompant le fichier log. Protéger avec un `DispatchQueue` ou un lock, ou utiliser `os_log` / `OSLog`.

12. **Logger.swift:69-70** — Durant la rotation, si `moveItem` échoue silencieusement (`try?`), puis `createFile` crée un fichier vide, les logs existants sont perdus sans avertissement. Logger le résultat du move (dans stderr au minimum, puisque le fichier log est en cours de rotation).

13. **AIUsagesTrackersApp.swift:13** — `Task { await pollerRef.start() }` est fire-and-forget. Si le poller échoue lors du premier poll, aucune UI ne reflète l'erreur. Ce n'est pas un bug critique mais un point d'attention pour l'observabilité.

14. **ClaudeCodeConnector.swift:53** — `URL(string: ...)!` force-unwrap. L'URL est un littéral donc le crash est improbable en pratique, mais un `guard let` serait plus défensif et cohérent avec le style du reste du fichier.

6. **UsagePoller.swift:49** — `pollOnce()` avec un tableau de connecteurs vide (`connectors: []`) → les entries seront vides, `guard !entries.isEmpty` déclenche un early return. **Pas testé.**

## Principe non-négociable — discipline des commits

**Tu commites AVANT de présenter le résultat à l'utilisateur, JAMAIS
après son approbation finale.** Dans cette phase : l'utilisateur a
déjà approuvé une *sélection* de recommandations → tu les appliques
→ tu commites → tu présentes le résultat → l'utilisateur choisit
ensuite entre "apply more", "re-run reviews" ou "ship it". Le
commit correspond à un état produit, pas à une validation finale.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception.**

**FIXUP — règle absolue pour cette phase :** chaque application de
recommandation doit produire des commits `--fixup` rattachés au commit
source qui a introduit le code modifié, PAS des commits indépendants
`review(axe):`. Procédure pour chaque bloc de fichiers modifiés :
```bash
# Trouver le commit source pour les fichiers concernés
git log --oneline origin/main..HEAD -- <fichier1> <fichier2>
# Le commit le plus récent qui a touché ces fichiers est le commit source
SOURCE_HASH=<hash du commit source>
git add <fichiers du bloc>
git commit --fixup=$SOURCE_HASH
```
Si un bloc touche des fichiers issus de **plusieurs** commits sources
distincts, crée **un fixup par commit source** (staging séparé par commit).
L'historique final doit être squashable proprement avec
`git rebase --autosquash`.

6. **UsagesFileManager.swift:30-34** — `withFlockExclusive` dans `update(with:)` ne retourne rien, mais si le flock échoue (retourne `nil` dans `withFlock`), l'update est silencieusement ignorée : les données reçues sont perdues sans aucun log au niveau appelant. Propager l'échec ou au minimum logger au niveau `update()`.

7. **UsagesFileManager.swift:100-102** — TOCTOU race sur le fichier lock : `fileExists` (L101) suivi de `createFile` (L102) n'est pas atomique. Un autre processus peut supprimer le fichier entre les deux. Utiliser directement `open(lockPath, O_RDWR | O_CREAT, 0o644)` qui crée atomiquement.

8. **UsagePoller.swift:29-35** — Le `Task` capture `[weak self]` mais l'actor `UsagePoller` n'est pas une classe — c'est un actor, qui ne supporte pas `weak`. Ce code compile mais `weak self` n'a aucun effet : l'actor ne sera jamais déalloué tant que la task tourne. `stop()` annule la task, mais si `stop()` n'est jamais appelé (ex. crash avant), la task et l'actor fuient.

9. **UsagePoller.swift:33** — `try? await Task.sleep(for: self.interval)` avale le `CancellationError`. C'est intentionnel (boucle while), mais après le sleep le `Task.isCancelled` est vérifié uniquement au prochain tour de boucle, ce qui signifie qu'un `pollOnce()` complet sera exécuté après l'annulation si le timing est défavorable. Vérifier `Task.isCancelled` immédiatement après le sleep.

10. **UsagePoller.swift:53** — `Loggers.app.log(...)` est utilisé au lieu de `self.logger`. Cela contourne l'injection de dépendance du logger, rendant cette erreur invisible dans les tests qui injectent un logger custom.

11. **Logger.swift:48-61** — `append()` n'a aucune synchronisation. `FileLogger` est marqué `Sendable` et peut être utilisé depuis plusieurs threads/actors concurrents. Deux appels concurrents à `append()` peuvent interleaver les écritures (seek + write non-atomique), corrompant le fichier log. Protéger avec un `DispatchQueue` ou un lock, ou utiliser `os_log` / `OSLog`.

12. **Logger.swift:69-70** — Durant la rotation, si `moveItem` échoue silencieusement (`try?`), puis `createFile` crée un fichier vide, les logs existants sont perdus sans avertissement. Logger le résultat du move (dans stderr au minimum, puisque le fichier log est en cours de rotation).

13. **AIUsagesTrackersApp.swift:13** — `Task { await pollerRef.start() }` est fire-and-forget. Si le poller échoue lors du premier poll, aucune UI ne reflète l'erreur. Ce n'est pas un bug critique mais un point d'attention pour l'observabilité.

14. **ClaudeCodeConnector.swift:53** — `URL(string: ...)!` force-unwrap. L'URL est un littéral donc le crash est improbable en pratique, mais un `guard let` serait plus défensif et cohérent avec le style du reste du fichier.

## Principe non-négociable — discipline des commits

**Tu commites AVANT de présenter le résultat à l'utilisateur, JAMAIS
après son approbation finale.** Dans cette phase : l'utilisateur a
déjà approuvé une *sélection* de recommandations → tu les appliques
→ tu commites → tu présentes le résultat → l'utilisateur choisit
ensuite entre "apply more", "re-run reviews" ou "ship it". Le
commit correspond à un état produit, pas à une validation finale.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception.**

**FIXUP — règle absolue pour cette phase :** chaque application de
recommandation doit produire des commits `--fixup` rattachés au commit
source qui a introduit le code modifié, PAS des commits indépendants
`review(axe):`. Procédure pour chaque bloc de fichiers modifiés :
```bash
# Trouver le commit source pour les fichiers concernés
git log --oneline origin/main..HEAD -- <fichier1> <fichier2>
# Le commit le plus récent qui a touché ces fichiers est le commit source
SOURCE_HASH=<hash du commit source>
git add <fichiers du bloc>
git commit --fixup=$SOURCE_HASH
```
Si un bloc touche des fichiers issus de **plusieurs** commits sources
distincts, crée **un fixup par commit source** (staging séparé par commit).
L'historique final doit être squashable proprement avec
`git rebase --autosquash`.
