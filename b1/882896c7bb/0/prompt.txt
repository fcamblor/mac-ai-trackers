## Discipline des commits

**Tu commites AU FIL de ton implémentation, un commit par bloc logique,
AVANT que le gate déterministe ou l'utilisateur ne valide quoi que ce
soit.** Le commit matérialise le travail ; la validation porte sur
l'état déjà committé.

**LANGUE — règle absolue : tous les messages de commit (sujet ET body)
sont rédigés en anglais, sans exception.**

## Ta mission cette itération

1. Identifie ce qui est déjà implémenté (via `git log`) vs ce qui reste
   dans le fichier de plan.
2. Implémente le prochain bloc logique (pas nécessairement tout —
   itérer est OK).
3. Écris des **tests solides** : cas nominal + cas limites + cas d'erreur.
   N'attends pas qu'une review downstream de couverture de tests
   rattrape les trous — tu possèdes la qualité de couverture ICI.
4. Commit AU FIL, un commit par bloc logique :
   ```bash
   git add <fichiers>
   git commit -m "<type>: <subject in English>" \
     -m "<WHY — retained trade-offs, discarded alternatives, technical decisions, in English>"
   ```
   Types : `feat:`, `test:`, `refactor:`, `fix:`, `chore:`.
   NE DÉCRIS JAMAIS le QUOI dans le body (le `git diff` s'en charge) —
   toujours le POURQUOI.
5. Avant de te déclarer prêt, lance toi-même le build + tests via bash.
