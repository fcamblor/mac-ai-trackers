# Review — Code quality

1. [VendorStatusBanner.swift:54] `.buttonStyle(.plain)` is a no-op here — the parent is a `VStack`/`HStack` with `.onTapGesture`, not a `Button`. Remove the dead modifier or convert the tap target to a `Button` (which would also gain proper accessibility for free).

2. [UsageModels.swift:225] `Outage.url` is a raw `String?` but the project's value-object rule says to wrap domain primitives with distinct identity. It is parsed into a `URL` at the use-site (VendorStatusBanner:50) anyway — consider a lightweight value object (or just `URL?` with a custom Codable) to push validation to decode-time rather than tap-time.

3. [UsageModels.swift:36-38] `vendors.flatMap` iterates a `Dictionary`, so `usages` returns entries in non-deterministic order. Callers currently cope (sort before display, keyed lookup in merge), but a future caller that assumes stable ordering will get bitten. A one-line `.sorted(by:)` on the outer iteration would make the contract explicit and cheap.

4. [UsageDetailsView.swift:29] `vendorsNeedingBanner(in:)` is recomputed on every SwiftUI body evaluation even when `store.outagesByVendor` hasn't changed. For a handful of entries this is fine, but if entry count grows, hoisting the result into a stored/derived property on the store (or a simple `let` outside the `ScrollView`) avoids repeated work inside the view builder.

5. [UsagesFileManager.swift:89-126] `merge` flattens to `[VendorUsageEntry]` then writes back through the `usages` setter, causing two full traversals of the vendor map. Operating directly on `vendors` sections (keyed merge per vendor) would be both more efficient and make the "outages are untouched" guarantee structural rather than relying on the setter's preservation logic.

- Error handling: # Review — Error handling

1. **UsageModels.swift:84-86** — `UsagesFile.init(from:)` silently falls back to an empty file when JSON contains neither `schemaVersion` nor `usages`. A corrupted or unexpected payload (e.g. `{"foo":1}`) is indistinguishable from a genuinely empty file. Consider throwing a `DecodingError` or at minimum logging a warning so data loss is diagnosable.

2. **UsageModels.swift:222-225** — `Outage.url` is `String?` but never validated at decode time. An upstream payload with `"url": "not a url"` passes through silently — the failure only surfaces later in the UI when `URL(string:)` returns `nil` on tap. Validate at the boundary (decode as `URL?` or validate the string) so bad data is caught early rather than silently swallowed at interaction time.

3. **VendorStatusBanner.swift:49-52** — `onTapGesture` silently no-ops when the URL is missing or unparseable. The user taps a banner and nothing happens, with no feedback. Consider disabling the tap gesture (or removing the clickable affordance) when there is no valid URL, so the UI doesn't look broken.

4. **UsagesFileManager.swift:66** — `guard var section = file.vendors[vendor] else { return }` in `updateIsActive` silently discards the request when the vendor section doesn't exist. If a connector registers an account and then immediately calls `updateIsActive` before the file is flushed, the active-account switch is lost with no warning. A `logger.log(.warning, ...)` here would make the race diagnosable.

5. **UsagesFileManager.swift:123-125** — The merge rebuilds the vendor map via the `usages` setter, which preserves outages from `existing`. But if `incoming` introduces a _new_ vendor that also has outages in `existing` (impossible today but fragile), the setter path won't carry those over because `vendors` is rebuilt from the setter's `newValue`. The implicit coupling between the setter's outage-preservation logic and the merge's `result.usages = usages` assignment is not obviously correct on first read — a comment or assertion (`assert(result.outagesByVendor == existing.outagesByVendor)`) would guard against future regressions.

6. **UsagesFileManagerTests.swift:423,462** — `try!` on `Data.write(to:options:)` in test setup. Per project rules (`SWIFT-ERROR-HANDLING`), `write` is a correctness-critical operation. Use `try` with the test's throwing context instead, so a file-system failure produces a clear test error rather than a crash.
