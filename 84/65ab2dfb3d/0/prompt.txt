# Review — Code quality

1. [Logger.swift:145] `DispatchQueue.main.sync` inside `resolveLevel()` will deadlock when `log()` is called from the main thread. The call path is: main-thread `log()` → `effectiveMinLevel` → `dynamicLevel()` → `resolveLevel()` → `DispatchQueue.main.sync { prefs.logLevel }` → deadlock. Consider using `Thread.isMainThread` to call synchronously when already on main, or restructure so that `resolveLevel` never does a sync hop to the current queue (e.g. cache the level value via an observation callback instead of reading it per-call).

2. [Logger.swift:36–59] The two `FileLogger` initializers duplicate the directory-creation block verbatim (6 lines each). Extract into a `private static func ensureDirectory(for filePath: String)` to remove the duplication — this is a real maintenance risk since a fix to one init could miss the other.

3. [UsagePoller.swift:56] `fixedInterval ?? .seconds(180)` — the `?? .seconds(180)` fallback is unreachable: the test init always assigns `fixedInterval = interval` (which itself defaults to `.seconds(180)`), and the production init requires `preferences`. The dead branch hides an implicit magic number; either remove the fallback with a `fixedInterval!` (provably safe) or make the default explicit via a named constant.

4. [GeneralSettingsView.swift:22–24] The `Slider` binding's setter `{ preferences.refreshInterval = RefreshInterval(clamping: Int($0)) }` truncates the `Double` via `Int(_:)`. Slider continuous updates produce values like `89.999…` which truncate to `89` instead of rounding to `90`. Use `Int($0.rounded())` instead.

5. [LoggingSettingsView.swift:7–9] `envOverrideActive` reads `ProcessInfo.processInfo.environment` on every view body evaluation. This value is immutable for the process lifetime — compute it once (e.g., `static let` or `@State` initialized in `init`) rather than per-render.
```
- Error handling: Now let me check a couple of specific areas more closely.# Review — Error handling

1. **`Logger.swift:53-58`** — The new `init(filePath:dynamicLevel:)` catches the `createDirectory` error but only prints to stderr and continues. If the directory does not exist, every subsequent `append` call will silently fail to write logs. Per `SWIFT-ERROR-HANDLING.md §1`, `createDirectory` must not be swallowed. Either propagate (make the init `throws`) or, at minimum, set a flag so later writes log to stderr as a fallback. Note: the existing `init(filePath:minLevel:)` on line 37–45 has the same pattern — the new init copied it.

2. **`Logger.swift:145`** — `DispatchQueue.main.sync { prefs.logLevel }` inside `resolveLevel()` will deadlock if this is ever called from the main thread (e.g., a `Loggers.app.log(...)` on the main queue). The logger's serial queue dispatches to main synchronously — if that serial queue callback is itself called from main, it's a classic `sync`-on-same-queue deadlock. Consider `DispatchQueue.main.async` with a cached last-known level, or guard `Thread.isMainThread` to read directly.

3. **`Logger.swift:128-131`** — `nonisolated(unsafe)` on `_preferences` relies on the comment's "written exactly once before any log call" invariant. There is no enforcement of this ordering — if any static logger (`Loggers.app`) is used at file scope or in another `static let` initializer before `AppDelegate.applicationDidFinishLaunching` runs, the read races with the write. Consider using an atomic or `os_unfair_lock`-protected wrapper instead of relying on launch ordering.

4. **`LaunchAtLoginService.swift:23-28`** — `setEnabled(_:)` throws the raw `SMAppService` error. The protocol declares `throws` which is good, and `GeneralSettingsView` catches it (line 47–48). However, the error surfaced is `error.localizedDescription` which for `SMAppService` failures is often vague (e.g., "The operation couldn't be completed"). Consider wrapping in a domain error with the `enabled` flag and underlying error for better diagnostics.

5. **`AppDelegate.swift:49-50`** — The reconciliation `Self.sharedPreferences.launchAtLogin = launchService.isEnabled` reads `SMAppService.mainApp.status`. If macOS returns `.notFound` (app not in Login Items database), `isEnabled` returns `false`. This is fine. But if `.requiresApproval` is the status, `isEnabled` also returns `false`, which may silently flip the preference to `false` even though the user did enable it in-app. Consider handling `.requiresApproval` distinctly — at least log it.

6. **`UsagePoller.swift:69`** — `try? await Task.sleep(for: sleepDuration)` — the `try?` here is acceptable (cancellation), but the `sleepDuration` comes from `currentInterval()` which hops to `@MainActor`. If the main actor is blocked (e.g., a long modal alert), this `await` will stall the poller. Low severity but worth noting.

7. **`GeneralSettingsView.swift:42-49`** — The toggle's `set` closure calls `launchAtLoginService.setEnabled(newValue)` and only updates the preference on success. Good error-path discipline. However, if `setEnabled` throws, the Toggle's binding `get` still returns the old (stale) value, so SwiftUI will visually snap the toggle back — but there's a brief flash. Consider using `@State private var launchAtLogin` seeded from preferences to give the UI a stable source of truth.

8. **`AppPreferences.swift:28-29`** — `defaults.integer(forKey:)` returns `0` for a missing key, which is treated as "return default". This is correct. But if a caller writes `refreshInterval = RefreshInterval(clamping: 30)` (the minimum), it stores `30`, and reading back gives `30` (not `0`), so the "return default" path is never hit for the minimum value. No bug, but worth a comment — the sentinel `0` conflates "never set" with an impossible clamped value.

Items #3–#5 are moderate-severity race/edge-case concerns. The rest are low severity.
